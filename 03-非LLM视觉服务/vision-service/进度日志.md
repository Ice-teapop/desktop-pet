# 非 LLM 视觉服务 — 进度日志

> 实时记录每一步的改动、验证结果与关键决策。新条目追加在底部。

## 步骤 1 — 可运行的服务骨架

- **改动**：FastAPI 工程骨架；按《接口契约 v1》立起 `/v1/extract`、`/v1/capabilities`、`/v1/health`；契约数据模型（`schema/contract.py`）；Bearer Token 鉴权 + 统一错误响应体；流水线骨架（仅校验图像可解码、返回占位块）。
- **验证**：8 个契约测试全过。
- **状态**：✅ 完成。

## 步骤 1 收尾 — 审查修复（第一批）

- **改动**：token 比较改常数时间（`hmac.compare_digest`）；启动安全自检（默认占位 token 拒绝启动，放在 lifespan）；`Image.open` 改上下文管理器；`config.targets` 与 `recognizers` 对齐（去掉 symbol）。
- **验证**：9 个测试全过（+1）。
- **状态**：✅ 完成。

## 步骤 1.5 — 审查剩余项

- **改动**：Content-Length 粗检；解压炸弹防护（`max_image_pixels`）；按 token 滑动窗口限流（429 + Retry-After）；校验错误不回显内部结构；结构化日志（不记画面内容）；`X-Request-Id` 清洗（限字符集 + 限长）；`bbox` 约束为 4 元素。
- **验证**：12 个测试全过（+3）。
- **状态**：✅ 完成。

## 步骤 2 — 预处理 + 文本 OCR + 文本转码

- **改动**：预处理（灰度 / 自动对比度 / 小图放大，`preprocess.py`）；可插拔 OCR 引擎（`ocr/`，已实现 Tesseract）；文本识别器 + 文本转码器；流水线串起「解码 → 预处理 → 文本识别 → 文本转码」；OCR 引擎不可用时优雅降级。
- **验证**：15 个测试全过（+3）；真实链路实测识别出文字。
- **决策**：OCR 引擎用 Tesseract、可插拔；预处理只用 PIL，不引入 OpenCV。
- **状态**：✅ 完成。

## 步骤 3 — 版面分析 + 区块分派 + 结构化合并

- **改动**：版面分析（规则版水平空白投影切区域，`layout.py`）；注册表（`registry.py`）；区块分派（按类型路由 + 坐标还原 + 转码，`dispatch.py`）；结构化合并（`assembly.py`）；`Recognizer.recognize` 签名改为吃 `Image`，坐标还原移到分派层。
- **验证**：19 个测试全过（+4）；真实链路实测上下两段文字切成 2 个区块。
- **过程 bug**：区域裁剪太紧、零边距导致 tesseract 识别不出 → 修复为裁剪时留 padding，并用实际裁剪原点还原坐标。
- **定位确认**：视觉服务只「提取直观信息」，不「理解」—— 理解交给 LLM。区域类型暂统一标 `text`，精确类型检测留到步骤 4（不做假精度）。
- **状态**：✅ 完成。

## 步骤 3 收尾 — 审查修复

- **改动**：版面分析加 `max_regions` 上限（防被构造的图打出大量 OCR 子进程）；`TesseractEngine.available()` 结果缓存（不再每请求 shell out）；`engine` / `analyzer` / `registry` 改模块级单例（启动建一次、请求间复用）。
- **验证**：21 个测试全过（+2）；实测多请求复用单例、`available()` 已缓存。
- **状态**：✅ 完成。

## 步骤 4a — 代码识别器

- **改动**：轻量分类器（`classify.py`，基于 OCR 文本内容判 text / code）；代码识别器（`recognizers/code.py`，OCR + 用行首 x 重建缩进 + 轻量语言判断）；代码转码器（`transcoders/code.py` → `{lang, text}`）；注册表登记 "code"；dispatch 接入分类。
- **验证**：27 个测试全过（+6）；真实链路实测代码截图 → 一个多行 code 块、lang=python、缩进已重建。
- **过程修复**：① 误用 `cp` 在 tests/ 下建了 `tests/src/`，已清理；② layout 把代码按行拆碎 → 改成两段式自适应切分（先找行、再按「行间距 > 行高 × 系数」才划区域边界），段落 / 代码块不再被拆碎。
- **已知限制**：代码区域会二次 OCR（分类一次、代码识别一次），后续可优化为复用；语言判断是轻量启发式；OCR 对小字保真度有限（属引擎能力，非流水线问题）。
- **状态**：✅ 完成。

## 步骤 4b — 表格识别器

- **改动**：网格线检测（`grid.py`，找水平/垂直长暗线，返回线的实际区间）；表格识别器（`recognizers/table.py`，用网格线切单元格、逐格 OCR、第一行作表头）；表格转码器（`transcoders/table.py` → `{headers, rows}`，`target="table-json"`）；版面分析对带网格的区域标 `table`；注册表登记 "table"。
- **验证**：32 个测试全过（+5）；真实链路实测表格截图 → 一个 table 块、`target=table-json`、4 格读对 3 格。
- **过程修复**：① `recognizers/table.py` 导入路径写错（`..config` → `...config`）；② 单元格 OCR 大半失败 —— 根因是 inset 太小、网格线残留进单元格，改成 `find_grid_lines` 返回线的实际区间、按真实线宽精确避让，单元格 OCR 改「裁到内容外接框 + 放大 + 补白边」。
- **已知限制**：只处理「有可见网格线」的表格（无边框表格交文本处理，不做假精度）；逐格 OCR 对极短小字仍可能漏（属 OCR 引擎保真度，真实截图正常字体会好很多）。
- **状态**：✅ 完成。

## 步骤 4c — 公式识别器

- **改动**：可插拔公式引擎（`formula/`，`FormulaEngine` 抽象 + `Pix2TexEngine` 实现 + 工厂）；公式识别器（`recognizers/formula.py` → LaTeX）；公式转码器（`transcoders/formula.py`，`target="latex"`，encoded 是 LaTeX 字符串）；`classify.py` 扩展出 formula 检测；`dispatch.py` 改成 text 区域可重判为 code / formula，且专项识别器无结果时优雅降级回文本；注册表登记 "formula"。
- **验证**：37 个测试全过（+5）；真实链路实测「公式引擎不可用时优雅降级为文本」生效。
- **未能验证的部分（如实说明）**：pix2tex 含 PyTorch、依赖过重，开发沙箱装不上 —— pix2tex 的实际推理路径（`Pix2TexEngine`）**未经端到端实测**，按官方 API 编写，需在部署服务器上验证（`pip install pix2tex` 后跑 `/v1/extract` 发公式图）。其余（引擎抽象、识别器、转码器、分类、注册表、优雅降级）已用假引擎 + 降级场景充分验证。
- **已知限制**：公式区域检测召回率有限 —— tesseract 不认数学符号，很多公式 OCR 出来不含数学符号会被漏判成普通文本。规则法的固有局限，不做假精度；要可靠检测需后续上检测模型。
- **状态**：✅ 完成（pix2tex 推理路径待服务器验证）。

## 步骤 4d — 审查修复（上线前）

- **审查**：用 careful-reviewer HIGH 密度审查三个视觉插件（code / table / formula），得 4 条 should-fix + 4 条 suggestion + 3 条 nit。
- **修复（4 条 should-fix）**：
  - **#1** `classify.py` 加 `math_density >= 0.10` 阈值，防含 ≥2 数学符号的长代码被误判为 formula；
  - **#2** `recognizers/formula.py` + `transcoders/formula.py`：`Block.raw_text=""` + `notes` 含 `latex_only` —— 守住契约 5.3「raw_text 是近似线性文本兜底」语义（pix2tex 不做 OCR 故无兜底）；
  - **#3** `FormulaEngine` 加 `warmup()`，`main.py` lifespan 启动时调用；`Pix2TexEngine.warmup` 失败强制 `_import_ok=False`（顺带修 #7：available 不验模型加载）；
  - **#4** `Pix2TexEngine._ensure_model` 加 `threading.Lock` 双检锁，防 FastAPI threadpool 并发首请求各加载一份 ~1GB 模型导致 OOM。
- **验证**：本地 41 测试 36 pass + 5 OCR skip；服务器 41 测试 40 pass + 1 降级测试自动 skip（公式引擎可用就跳）。补了 4 条回归测试。
- **状态**：✅ 完成。

## 部署上线 — Vultr Melbourne 2GB（2026-05-15）

- **SSH 加固**：本地 `~/.ssh/config` 加 `deskpet-prod` 别名；服务器 `/etc/ssh/sshd_config.d/00-deskpet-hardening.conf` 关密码登录、改 `PermitRootLogin prohibit-password`、关键盘交互后门，验证密码登录被拒。
- **一键发布脚本**：`deploy/deploy.sh` —— `rsync --delete` 同步代码 → `chown deskpet:deskpet` → `systemctl restart` → `/v1/health` 自检；以后 `./deploy/deploy.sh` 即发布。
- **服务器侧**：apt 装 tesseract 5.5 + chi_sim + eng；服务器自带 5.3GB swap（超 2GB 文档要求）；`useradd deskpet`（nologin，非 root）；rsync 代码到 `/opt/deskpet/vision-service/`；`.venv` + `pip install -r requirements.txt` + `pip install pix2tex`；生成 64 字符 token 写 `/etc/deskpet-vision.env`（chmod 600）；装 `deskpet-vision.service` + `enable --now`。
- **关键踩坑**（已记入 `.careful-coder/notes/`）：
  - `/tmp` 是 tmpfs 819MB，PyTorch wheel 532MB 直接爆 —— `TMPDIR=/var/tmp` 走真实磁盘解决；
  - 装完 pip 后必须再 `chown` 一次，否则新装的文件归 root，deskpet 写不了模型 checkpoint，预热失败。
- **验证清单**：
  - ✅ `/v1/health` `ok:true`
  - ✅ `/v1/capabilities` 返 4 识别器 + 3 target
  - ✅ `/v1/extract` 真实 OCR 识别测试文字
  - ✅ pytest 服务器端 40 pass + 1 skip
  - ✅ **pix2tex 模型预热完成**（97.4M weights + 18.5M image_resizer 下载就位，LatexOCR 实例化成功 —— 这是开发沙箱无法验证的核心）
  - ✅ `to_latex()` 真实推理产出 LaTeX
  - ✅ 日志合规：只记 region_id / frame_seq / 延迟 / 区块数，不含画面内容
- **实测资源**：FastAPI + tesseract + pix2tex 预热完成后 **1.24 GB** 常驻（部署文档预估 1.3–1.8 GB 下限），swap 用 ~250MB 兜底。
- **状态**：✅ 完成。
