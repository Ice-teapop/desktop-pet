careful-coder (cc) 优缺点

合并 M0–M3 + v0.4.x 观察。打 ★ 的是 v0.4.x 新增。

✓ 做得好

  • typed error 归类（M2-1）—— 用 SDK 提供的 error class 而非字符串匹配，体现对 SDK 设计的理解
  • race 防御的架构级清理（M2-2）—— 不是补丁式，引入 chatTurnToken + 串行化 queue + 真 AbortController 三层防御
  • migration 思路（M3-3）—— productName 改名后用 fs.rename 原子搬迁 + 删 legacy 防"被遗忘的明文 key"
  • 可选关闭设计（M3-3）—— 监视类 feature 默认开但加托盘 checkbox，尊重 privacy
  • ★ 多 agent 跨视角设计（v0.4.x）—— 4 agent 联合给出 4.6s 巫师帽事故 sequence，单 agent 想不到的 keyframe 节奏
  • ★ release pipeline 熟练度（v0.4.x）—— gh CLI / tag 删除重发 / draft → latest 一气呵成
  • ★ 用 user 提供的资源 + 借鉴授权（v0.4.x）—— pet-forge MIT skill 接入快，老 Clawd 动画 keyframes 借鉴获明确授权

✗ 做得不够

  • 顶部 docstring 同步反复 4 次（M2-2 → M3-3-D）—— cr 每次都标 ✨，cc 修 3 次后第 4 次又忘
  • 资源边界 audit 缺失（M3-3）—— user 说"动画单一"才意识到 themes/clawd-dev/ 只有 4 SVG，上游有 45 SVG + 23 GIF
  • 默认参数 conservative（M3-3）—— POLL_MS=5000 是直觉值，user 反馈"立刻"后才推 500ms，再推 Swift binary
  • 过度防御性参数（M3-3-D）—— <img key={gifUrl}> 想"让 GIF 从第一帧播"，实际 Chromium 自动重解码，反而 1 帧透明闪烁
  • 死代码遗留（M3-3）—— pet:event:click IPC 是 M1 demo 用的，cr 找出才清理
  • fix-first 反模式（M3-3-G 点不开 bug）—— 6 轮全凭猜 build+install 才转 diagnose 加 log
  • ★ 用 Write 覆盖代替 Edit（v0.4.14）—— bump version 时把 74 行 package.json 截成 6 行垃圾 stub，CI 全挂 → release 0 artifact → 必须重发 v0.4.15。灾难性
  • ★ 沉默失败 4 轮（shadow rAF）—— CSS animation 覆盖 inline transform，cc 改了 4 轮参数都没意识到根因
  • ★ 同类沉默失败 2 例（eye CSS、shadow CSS）—— "渲染层优先级"没成为 cc 的 default checklist
  • ★ 文件定位错（v0.4.14）—— monorepo 双 README，编辑了 subdir 那个，GitHub 实际显示根的
  • ★ 美术质量 0/4 通过—— 已存 memory feedback「不要替用户手写 SVG 美术稿」
  • ★ 缩放方向理解错—— user 要"缩小 pet"，cc 改了 viewBox 等于"放大对话框"，需要 user 再纠正

careful-reviewer (cr) 优缺点

cr 在本次 v0.4.x 没显式 invoke，所以新数据较少；主要数据来自 M0–M3 阶段。

✓ 做得好

  • 找到真正的 race（M2-2 chatHistory）—— 追时序找出 resetKey + in-flight stream 的链，cc 完全没想到
  • path drift 风险预警（M3-1）—— productName 改名时立刻标 🚫 BLOCKING，user 后来实测验证
  • safe-but-not-clean 关注—— migration 用 copyFile 留 legacy 时 cr 要求改 rename 删源
  • 跨 dimension 状态 audit（M3-2）—— setModel 时发现 thinking 状态没自动结束 + 流式 cursor-blink 卡死

✗ 做得不够

  • 同类问题反复提示但没强制整改—— docs-drift 4 次都 ✨ nit，没升 ⚠️
  • speculative 标记偶尔过谨慎—— did-finish-load 推 key:state 标 [speculative]，user 没遇到但 cr 仍要求加防御
  • 少数 race 实际不可复现—— savePreferences 连切丢需 200ms 内连切 4 次，加 debounce 可能 over-engineering
  • 推 fix 前不强制 reproduce（M3-3-G 点不开 bug）—— 推测 transparent hit-test 是根因，user 实测仍失败，第二轮才转 diagnose
  • ★ 本次 v0.4.x 没自动 invoke—— release engineering / 主题切换没走 cc → cr 复审环节，package.json 灾难才漏过

多 agent 协作（三方会谈 / 4 专家联合）适用边界

M3-3 引入三方会谈、v0.4 引入 4 专家联合设计动画。两次实践的边界总结。

✓ 适用场景

  • 跨 domain 诊断（视觉 UX / 技术实现 / 调度逻辑 / macOS 系统集成 / LLM 调用层）— 单 agent 看不到的盲区
  • 设计型工作（动画 keyframe 节奏 / 状态衔接 sequence）— 多视角 brainstorm 质量高
  • 数字化感知 — "感觉慢" 翻译成具体延迟段落（"P50 ~620ms 主要来自 600ms debounce"）
  • 边界 quirk 发现 — 如 Swift binary 用错 NotificationCenter，单 agent 几乎不会主动深挖 Cocoa 边界

✗ 不适用场景

  • 深度技术陷阱（React 18+ functional setState quirk、CSS animation 覆盖 inline transform）— 单 agent 长经验比三方都看一遍更靠谱
  • 共识陷阱 — 三方一致推荐的方案要小心，他们可能都基于同一个错误假设。user 实测仍失败时要怀疑共识本身
  • fix unverified 比 no fix 更糟 — 浪费 build+install 周期 + user 信任。三方设计要带 reproduce 验证
  • 替代 diagnose 模式 — 三方会谈不能代替加 log + 真实观测数据

综合改进建议（v0.4.x 新增 ★）

给 cc

  • 资源 audit 作为 Step 2 标配 — 先 ls / grep 边界，不假设默认值
  • 顶部 docstring 同步进 self-check checklist
  • 默认参数多想一步 "user 会感知吗"
  • 加防御性参数前问 "不加会出什么 bug"
  • 每次新加 IPC / state 时清理之前的遗弃代码
  • diagnose log 永远用 string concat，不用 object 序列化
  • fix 失败 2 轮内必须转 diagnose 模式（加 log）
  • React event 涉及 closure 副作用前，先区分 native vs React event；native 用 ref
  • ★ 编辑已有文件永远 Edit，不用 Write（v0.4.14 灾难根因）。Write 仅用于真正的新文件
  • ★ 编辑后必须验证文件大小没异常缩水 — wc -l 或 git diff --stat 看行数变化
  • ★ 渲染层 CSS animation vs inline style 优先级 作为 default checklist — 看到 inline transform 没生效先 grep CSS animation 同元素
  • ★ monorepo README 永远确认根目录文件（GitHub 显示根 README，不是 subdir）
  • ★ 美术 / sprite 资产不替 user 出（已存 memory feedback，不重复）

给 cr

  • 同类问题连续 N 次 ✨ → 自动升 ⚠️，逼 cc 系统修
  • 每个 ⚠️ 加 "user 视角可见？" 标签
  • [speculative] 数 > 实测 finding 时主动节制
  • cc 加冗余 catch / key / try 时 cr 标 ⚠️ "过度防御"
  • 推 fix 前评估 reproduce cost，高的话要求 diagnose log 一起 ship
  • fix 失败 N 轮后主动 push cc 转 diagnose 模式
  • ★ release 类操作（tag / push / 整文件覆盖）必须 invoke cr — v0.4.14 灾难就是 release 阶段没走 review
  • ★ "用 Write 覆盖已有文件" 自动 ⚠️ BLOCKING — 强制改 Edit 或要求 cc 解释为什么完全重写

给工作流

  • 每个 milestone 完工做一次 retrospective
  • 维护 lessons-learned 文档（就是 cc-cr-retrospective.md），跨 milestone 累积
  • user 反馈 "为什么 X" 时第一步问 "我假设了什么没验证"
  • 三方会谈适合 cross-domain 诊断，不适合 framework-specific quirk
  • 三方共识失败时怀疑共识本身，转 diagnose
  • ★ retrospective 文档要持续追加新事件，否则 v0.4.x 这种新增 cluster 会重复犯
  • ★ 每个 release 必须先本地 npm install --dry-run / git diff --stat 验证 — v0.4.14 本地一秒就能发现 package.json 缩成 6 行
  • ★ CI failed 后第一步是看 log，不是猜（本次按这点做对了，所以从失败到 v0.4.15 重发只用了 ~10 分钟）

结论 — cc / cr 的总评

维度表现说明
整体加速效果显著正收益M0→M3 24 小时交付原本估 1 周；v0.4 主题切换 1 夜完成 21 sprite 全替换
routine 代码任务≈ 85% 一次过架构思维 + SDK 用法 + race 防御达到资深工程师水准
渲染层沉默失败≈ 15% 一次过CSS-vs-inline 优先级、React-vs-native event 路径 — 单 agent 经验缺口
